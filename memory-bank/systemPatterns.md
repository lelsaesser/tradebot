# System Patterns

This document describes the system architecture, key technical decisions, design patterns, component relationships, and critical implementation paths.

## Architecture

The application follows a modular, component-based architecture built on the Spring Framework. The core logic is orchestrated by a central `Scheduler` component that triggers various tasks at scheduled intervals. These tasks are designed to be self-contained and resilient, with a `RootErrorHandler` providing a safety net for unhandled exceptions.

## Key Components

-   **`Scheduler`:** The heart of the application, orchestrating all scheduled tasks. Includes separate schedulers for stock and crypto monitoring with independent polling intervals. **Stays a thin cron-definition layer**: each `@Scheduled` method is a one-line `rootErrorHandler.run(tracker::method)` delegation plus a completion log line. Business logic (message formatting, data composition, branching) lives in dedicated tracker/service classes — `Scheduler` carries no domain logic of its own. Each `@Scheduled` method has a paired `manual*` variant (using `runWithStatus`) for `DevJobController` and Telegram triggers; both delegate to the same underlying tracker method.
-   **`SymbolRegistry`:** Unified `@Service` bean that owns all tracked symbols — ETFs (sector, thematic, benchmark as hardcoded constants) and individual stocks (JSON-loaded, dynamically add/remove via Telegram). Replaces the former split between `SectorEtfRegistry` (static) and `StockSymbolRegistry` (service). Methods: `getAll()`, `getAllEtfs()`, `getBroadSectorEtfs()`, `getThematicEtfs()`, `getStocks()`, `isEtf()`, `isSectorEtf()`, `fromString()`, `addSymbol()`, `removeSymbol()`.
-   **`SymbolManagementService`:** Shared service for add/remove of tracked symbols, used by both the Telegram `/add` / `/remove` command processors and the dashboard REST API. On remove: clears `SymbolRegistry` + `target_prices` directly, then fans out to every `SymbolLifecycleListener` bean to clean per-symbol state (RSI/RS/ROC state, OHLCV cache, accumulation streaks, ignored-symbols flags, newly-added-symbol queue, apex performers, insider transactions). See the **Symbol Lifecycle Listener** pattern below. End-user feedback is unchanged — cleanup is invisible from Telegram/UI, log-only on failure.
-   **`DailyPriceProvider`:** Unified data access layer for daily closing prices. Tries OHLCV (Twelve Data) first, falls back to Finnhub. Same `findDailyClosingPrices(symbol, days)` signature. Used by EmaService, BollingerBandService, RelativeStrengthService, MomentumRocService. OHLCV reads transparently cached via the `CachingOhlcvRepository` decorator (per-`(symbol, days)` entries, invalidated on every write path). Intraday quote append (the "today" data point on the OHLCV-hit path) reads from `LivePriceCache` — supports both domestic (Finnhub-written) and international (Yahoo-written) symbols.
-   **`*PriceEvaluator`:** A set of components (`FinnhubPriceEvaluator`, `CoinGeckoPriceEvaluator`, `YahooPriceEvaluator`) responsible for fetching and evaluating prices from different APIs. Finnhub and Yahoo evaluators write to the shared `LivePriceCache`.
-   **`LivePriceCache`:** Shared `@Service` bean holding the latest known price per symbol. Internal storage is `ConcurrentHashMap<String, PricedAt>` (public nested record carrying price + write timestamp). Public API: `put`, `get` (returns `Double`, hot path for callers needing only the price), `getEntry` (returns `Optional<PricedAt>` for callers needing the timestamp), `getAll` (price-only snapshot), `evictStale`. Both `FinnhubPriceEvaluator` (US stocks) and `YahooPriceEvaluator` (international stocks) write to it. Consumers (`PullbackBuyTracker`, `DailyPriceProvider`) read from it. Replaced the former `FinnhubPriceEvaluator.lastPriceCache` field. Stale entries (>24 hours) are evicted by `evictStale()`, called from `Scheduler.periodicMaintenance()` as a safety net against persistent values from a downed exchange or failing data source.
-   **`RsiService`:** Core service for RSI calculations. Manages historical price data, calculates RSI values, detects market holidays.
-   **`RsiPriceFetcher`:** Dedicated component for fetching historical price data for RSI calculations.
-   **`InsiderTracker`:** Tracks and reports insider trading activities. Excludes ETFs and international symbols (Finnhub free tier doesn't support non-US listings).
-   **`SectorRotationTracker`:** Tracks industry sector performance from FinViz.
-   **`SectorRotationAnalyzer`:** Statistical analysis component that detects sector rotation signals using Z-Score analysis.
-   **`RelativeStrengthTracker`:** Tracks stock performance relative to SPY benchmark using 50-period EMA crossover detection.
-   **`SectorRelativeStrengthTracker`:** Monitors sector ETF performance vs SPY benchmark. Real-time RS crossover alerts + daily summary. Uses `SymbolRegistry.getAllEtfs()` and `getThematicSymbols()` for report section splitting. Daily summary also appends a "stocks outperforming sector leader" section: identifies the top-ranked ETF (by RS-vs-SPY pctDiff descending) and lists tracked stocks whose RS-vs-leader pctDiff is positive, top 10 sorted descending, with total qualifying count. Uses `RelativeStrengthService.getCurrentRsResult(symbol, leader)` (benchmark-parameterized). Single Telegram message — section is appended, not a separate send.
-   **`MomentumRocService`:** Calculates Rate of Change (ROC) momentum and detects zero-line crossovers.
-   **`SectorMomentumRocTracker`:** Real-time sector ETF momentum analysis using ROC10/ROC20 values.
-   **`TailRiskService`:** Calculates excess kurtosis and skewness from daily price changes to detect fat tail risk. Takes a `TailRiskWindow(lookbackCalendarDays, minDataPoints)` record on `analyzeTailRisk(symbol, displayName, window)` — the calendar lookback is the oversized fetch window, `minDataPoints` is the actual statistical sample-size requirement. Sample is truncated to exactly `minDataPoints` (latest rows) for stable kurtosis/skewness comparability across runs. Returns `Optional.empty()` when fewer than `minDataPoints` prices are available. The two-value design follows the codebase convention used by EmaService/VfiService/RelativeStrengthService — fixed in #472 after a single conflated `lookbackDays` parameter caused the analysis to always reject (35-calendar-day query yields ~24 trading-day rows; `rows < 35` always failed). **Estimators:** uses Fisher-Pearson G2 for kurtosis and G1 for skewness via `org.apache.commons.statistics.descriptive` (1.3+) — bias-corrected sample formulas that are industry standard in quantitative finance (Pandas, NumPy `bias=False`, Excel `KURT`/`SKEW`, Basel-FRTB). Previously used the biased method-of-moments formulas (`m₄/m₂²`, `m₃/m₂^(3/2)`) which systematically under-read kurtosis and skewness at small n (~8% downward bias at n=25). Replaced in #433. The `calculateKurtosis(List<Double>)` method preserves the raw-kurtosis (normal=3.0) contract by adding 3.0 to Commons Statistics' excess-kurtosis return value; size guards (`< 4` / `< 3` returning 3.0/0.0 sentinels) and the zero-variance guard are preserved as NaN-prevention safety nets that protect against `TailRiskLevel.fromExcessKurtosis(NaN)` misclassifying as EXTREME.
-   **`TailRiskTracker`:** Monitors sector ETFs for elevated tail risk with directional context. Runs analysis twice per scheduler tick (`SHORT = TailRiskWindow(50, 25)` short window, `LONG = TailRiskWindow(400, 252)` long window — calendar-day fetch oversized, min-data-points captures the trading-day intent). Alerts: up to two independent Telegram messages per run, sent in stable order (35d first, 252d second), each tagged with `` `[35d]` `` / `` `[252d]` `` in the title. Daily report: single combined Telegram message with two tagged sections (`[35d window]` / `[252d window]`). Emits a `TAIL_RISK_COMPARE` INFO log line per symbol per daily run for offline grep-based analysis (issue #336 comparison period). **Formula boundary:** log lines emitted before PR #433-followup used the biased population formula; lines from #433-followup onward use Fisher-Pearson G2/G1. The two regimes are not directly comparable — at SHORT window, new values are ~1.08× higher (kurtosis) and ~1.15× larger in magnitude (skewness) than equivalent historical values; LONG window differences are < 1%. When analyzing the comparison period, filter log lines by merge date.
-   **`StatisticsUtil`:** Shared utility class providing `mean()`, `standardDeviation()`, `zScore()`, `percentileRank()`, `populationStdDev()`, `percentile()`, `calculateEma()`, `calculateRocValue()`, and `roundTo2Decimals()`.
-   **`BollingerBandService`:** Calculates Bollinger Bands (20-period SMA ± 2σ), %B positioning, bandwidth, and bandwidth percentile for squeeze detection.
-   **`BollingerBandTracker`:** Orchestrates Bollinger Band analysis across all sector ETFs and tracked stocks. Uses `SymbolRegistry.getAllEtfs()` for sectors and `SymbolRegistry.getStocks()` for individual stocks.
-   **`EmaService`:** Calculates EMAs (9, 21, 50, 100, 200 day) for a symbol. Calculates every EMA for which enough data exists (minimum 9 data points).
-   **`EmaTracker`:** Orchestrates daily EMA report across all tracked stocks via `SymbolRegistry.getAll()`.
-   **`VfiService`:** Calculates Volume Flow Indicator from OHLCV data (130-day lookback, 5-period signal line EMA). Returns `Optional<VfiAnalysis>`.
-   **`VfiTracker`:** Orchestrates daily RS+VFI combined report. Iterates `SymbolRegistry.getAll()`, classifies each symbol as GREEN (RS↑ + VFI↑), YELLOW (mixed), or RED (RS↓ + VFI↓) via `CombinedSignalType`. Sends via `TelegramGateway.sendMessage()` at 9:00 CET pre-market.
-   **`PullbackBuyTracker`:** Real-time EMA pullback buy alerts. Two entry points so the scheduler can gate domestic vs. international independently: `analyzeDomestic()` runs every 5 min during NYSE hours (inside `stockMarketMonitoring`), `analyzeInternational()` runs every 5 min unconditionally (after `yahooPriceEvaluator::evaluatePrice` so `LivePriceCache` is fresh) and skips per-symbol via `MarketStatusService.isExchangeOpen()`. This closes the prior gap where KRX stocks were never evaluated during their session and XETRA only got the 09:30–11:30 ET overlap. Detects stocks below EMA 9/21 but above EMA 50/100/200, with positive RS and VFI. Reads live prices from `LivePriceCache` (no separate API calls). Per-stock Telegram alerts with 8-hour cooldown via `IgnoreReason.PULLBACK_BUY_ALERT`. Alerts are tagged with a 🏆 _Apex performer_ highlight when the symbol is in `apex_performers` (set populated by the daily sector RS summary).
-   **`AccumulationDetectionTracker`:** Daily (10:00 CET) institutional accumulation detection. Identifies stocks where EMA9 < EMA21 (weak price) but VFI > 0 and VFI > signal (bullish volume). Sends consolidated Telegram alert with streak counter showing consecutive signal days (persisted in `accumulation_streaks` SQLite table). Streak annotation omitted for day-1 signals.
-   **`MarketHolidayNotifier`:** Owns the daily 8:00 CET (Mon-Fri) "Markets closed today" Telegram message. Single public method `sendDailyReport()`. Combines NYSE holiday data from `MarketStatusService.getTodayHoliday()` (early-close metadata included as "early close 13:00 ET") with the per-exchange international holiday map from `MarketStatusService.getTodayInternationalHolidays()` into one consolidated message; sends nothing when no exchange is closed today. Owns the `Exchange → flag emoji` mapping. Extracted from `Scheduler` (#504) so the scheduler stays a thin cron-definition layer.
-   **`TreasuryTracker`:** Monitors US Treasury macro signals via FRED API and emits a daily Telegram macro report at 08:30 CET MON-FRI. Tracks four series: `T10Y3M` (10Y-3M spread) and `T10Y2Y` (10Y-2Y spread) as recession-warning signals (classified via `YieldCurveSpreadLevel`, alerts on transitions into/out of `INVERTED`/`DEEPLY_INVERTED`), plus `DFII10` (10Y real yield, `RealYieldLevel`) and `THREEFYTP10` (10Y term premium, `TermPremiumLevel`) as report-only macro context. Single public method `checkAndAlert()` runs the full cycle: fetch 7-day window per series, take most-recent observation, classify, compare against persisted state via `TreasuryIndicatorStateRepository`, emit transition alert (separate Telegram message) on inversion crossings, then send the daily report. Partial-data resilient — one series failing renders as "—" in the report without suppressing alerts on the others. Band thresholds anchored to NY Fed Holston-Laubach-Williams r\* (DFII10) and NY Fed ACM (THREEFYTP10) literature. First-run alert suppression comes from row-absence in the state table (no prior row → skip alert, then INSERT a row with `initialized=true`); the `initialized` flag itself matches the `momentum_roc_state` convention as a hatch for manually-seeded rows. Required FRED ToS attribution footer in every report. Added in #516.
-   **`OhlcvFetcher`:** Fetches daily OHLCV data from Twelve Data for all tracked symbols (ETFs + stocks via `SymbolRegistry.getAll()`). Backfill: 400 data points. Refresh: 5 data points. 8-second delay between requests. Also provides `backfillSymbols(List<String>)` for on-demand backfill of newly added symbols.
-   **`OhlcvBackfillService`:** Processes newly added symbols that need OHLCV data. Reads from `newly_added_symbols` queue table (populated by `/add` command). Filters out symbols no longer tracked before fetching. Batch size: 10. TTL: 24h (expired entries logged and removed atomically via `DELETE ... RETURNING`). Runs in `periodicMaintenance`.
-   **`TelegramClient` / `LocalTelegramGateway` / `TelegramMessageProcessor`:** Telegram integration is profile-aware. All sending components inject `TelegramGateway` interface.
-   **`TelegramCommandDispatcher`:** Routes incoming commands to appropriate processors using the Command pattern.
-   **`TargetPriceProvider`:** Manages the watchlist of symbols to be monitored.
-   **`RootErrorHandler`:** Centralized error handler wrapping all scheduled tasks. `run()` preserves fire-and-log; `runWithStatus()` adds boolean success/failure for dev triggers. Logs caught exceptions via `log.error` only — Telegram error/exception reporting was removed in #469 (it was the largest secret-leak surface in the codebase, and journald + `scan-logs.sh` provide the diagnostic visibility it once supplied).
-   **`MarketStatusService`:** Single source of truth for market open/holiday state. Two distinct holiday caches: NYSE (Finnhub-sourced, with early-close metadata via `MarketHoliday`) and international exchanges (Enrico-sourced, `Map<Exchange, Map<LocalDate, String>>`). Both caches loaded at `@PostConstruct` and retried every 5 min by `@Scheduled` until populated; per-exchange `loaded` flags so one flaky country code doesn't block the others. Public API: `isMarketOpen(ZonedDateTime)` (NYSE, holiday-aware with early-close handling), `isExchangeOpen(String symbol)` (resolves ticker→exchange via `Exchange.fromTicker`), `getTodayHoliday()`, `getTodayInternationalHolidays()`. Per-exchange trading hours (incl. JPX morning/afternoon split with lunch break) live as constants in this service. `Exchange.getExtras()` provides a `Set<MonthDay>` overlay of exchange-specific full-day closures Enrico doesn't cover (e.g. Heiligabend on XETRA).
-   **`ApiRequestMeteringService`:** Per-provider request counter (Finnhub, CoinGecko, TwelveData, Yahoo). `AtomicInteger` map is the in-memory source of truth; `SqliteApiMeteringRepository` is crash-recovery persistence. Public API: `incrementXxxRequests()` (called from each client), `getXxxRequestCount()`, `getCurrentMonth()` / `getPreviousMonth()` (YYYY-MM), `flushCounters()` (called every 10 min from `Scheduler.periodicMaintenance` and on shutdown via `@PreDestroy`), `resetCounters()`, `sendMonthlyUsageReport()`. The monthly report builds and sends the previous-month Telegram summary (or skips Telegram when no usage was recorded), then resets counters — owns this end-to-end as of #504 (previously composed in `Scheduler`). `TelegramGateway` is a constructor dependency.
-   **`SecretRedactingTurboFilter`:** Logback `TurboFilter` registered globally via `src/main/resources/logback-spring.xml`. Scans every log event's format string, params (via `String.valueOf`), and throwable cause-chain `getMessage()` for known secret shapes (currently the Telegram bot token shape, regex `\d{8,10}:[A-Za-z0-9_-]{35}` from trufflehog's industry-standard pattern). On match returns `DENY` (drops the event) and emits a sanitized breadcrumb tagged with a bypass marker so the filter ignores its own output (recursion guard). Cycle-safe via `IdentityHashMap` on cause-chain walking. Added in #470 as defense-in-depth after #464; complements the `RootErrorHandler` Telegram removal in #469. Known blind spot: rendered stack frames (TurboFilter runs pre-event-creation), accepted residual risk since `Throwable.getMessage()` covers the only known concrete leak vector in this codebase.
-   **`DevDataSeeder`:** `dev`-only startup seeder that populates SQLite quote history (via `PriceQuoteRepository.saveAll()`), OHLCV data (400 days), RSI/RS/ROC state. Uses `JdbcTemplate` for direct table operations (DELETE, COUNT queries).
-   **`DevJobController`:** `dev`-only manual trigger surface. 19 individual endpoints + 1 composite `run-all` endpoint. Individual endpoints return HTTP 200/500. `run-all` orchestrates 4-phase smoke test execution.
-   **`DashboardEventPublisher`:** Spring service owning the active SSE emitter registry. `register(SseEmitter)` adds emitters (add-first, then wire callbacks to close registration race). `publish(String eventType, Object payload)` fans out `DashboardEvent` to all active emitters; catches `IOException` at warn (client gone, expected) and other `Exception` at error (unexpected bug). Emits a heartbeat `ping` event every 30 s on a dedicated single-thread `TaskScheduler` (`dashboardHeartbeatScheduler`) to isolate from long-running scheduled jobs.
-   **`SseController`:** REST controller at `GET /api/v1/events`. Creates `SseEmitter` with 8-hour safety timeout and delegates registration to `DashboardEventPublisher`. Active in all profiles (no `@Profile`).
-   **`DashboardConfig`:** `@Configuration` in `web/dashboard` package. Declares `dashboardHeartbeatScheduler` bean (`ThreadPoolTaskScheduler`, pool size 1) used exclusively by the SSE heartbeat.

## External API Clients

-   **`FinnhubClient`:** Stock prices and insider transactions. API key required.
-   **`CoinGeckoClient`:** Cryptocurrency prices. No auth.
-   **`FinvizClient`:** Web scraper using JSoup for industry performance. No auth.
-   **`TwelveDataClient`:** Fetches daily OHLCV data. API key required. 8 req/min rate limit. Metered via `ApiRequestMeteringService`.
-   **`YahooFinanceClient`:** Fetches daily OHLCV for international stocks (German/Korean/French/Japanese/Swedish) AND intraday price quotes (via `meta.regularMarketPrice`). No auth. Transport: `java.net.http.HttpClient` with `User-Agent: Mozilla/5.0`, 10s connect timeout, 15s per-request timeout. Verified in production over a 2-week window after #435; the legacy ProcessBuilder + curl path was removed in #457 once the original "Yahoo blocks Java HTTP clients via TLS fingerprinting" hypothesis was empirically disproven (no SSL/TLS failures observed). 3s delay between calls. Metered via `ApiRequestMeteringService`. Failures throw `YahooFetchException` (caught silently in OhlcvFetcher and YahooPriceEvaluator — no Telegram alert).
-   **`FredClient`:** Fetches US Treasury macro time series from the FRED® (Federal Reserve Economic Data) API for `TreasuryTracker`. Single endpoint: `/fred/series/observations`. Auth: single 32-char API key as `api_key` query param (free, email signup). Used to fetch four series: `T10Y3M`, `T10Y2Y`, `DFII10`, `THREEFYTP10`. The `"."` no-data sentinel is filtered at the wire→domain boundary (rows for weekends/holidays/pending publication carry it). Returns empty list + WARN log on any failure — matches `EnricoClient` shape. Not metered (free, ~4 req/day total, matches Enrico precedent). FRED ToS requires attribution in any consumer-facing rendering ("not endorsed by FRBSL") — surfaced by `TreasuryTracker` in the report footer, not here. Added in #516.

## Data Persistence Components

-   **`SqlitePriceQuoteRepository`:** Historical Finnhub price quotes via `JdbcTemplate`. Used as fallback for DailyPriceProvider. Supports batch insert via `saveAll()`.
-   **`SqliteMomentumRocRepository`:** Momentum ROC state via `JdbcTemplate` (previous ROC values for crossover detection).
-   **`SqliteOhlcvRepository`:** Twelve Data daily OHLCV data via `JdbcTemplate` in `twelvedata_daily_ohlcv` table. Primary source for DailyPriceProvider and VfiService. Batch insert via `BatchPreparedStatementSetter`.
-   **`CachingOhlcvRepository`:** `@Primary` decorator over `SqliteOhlcvRepository`. In-memory cache keyed by `(symbol, days)`, populated on cache miss via `ConcurrentHashMap.computeIfAbsent`, stored as immutable copies (`List.copyOf`). Invalidates entries for touched symbols on `saveAll(records)` (extracts distinct symbols from the batch) and `deleteBySymbol(symbol)`. All `OhlcvRepository` consumers receive the decorator transparently — they don't know data came from cache vs. SQLite. Caches empty results too; the same write-path invalidation hook covers the empty-then-populated transition for newly-added symbols.
-   **`SqliteIgnoredSymbolRepository`:** Per-symbol alert suppression via `JdbcTemplate` with reason codes and optional alert thresholds.
-   **`SqliteApiMeteringRepository`:** API request counters per provider via batch `INSERT OR REPLACE`. Flushed periodically by Scheduler's `periodicMaintenance()` (every 10 min) and on shutdown (`@PreDestroy`). `AtomicInteger` map is the in-memory source of truth; SQLite is crash-recovery persistence.
-   **`SqliteAccumulationStreakRepository`:** Accumulation streak data (consecutive signal days per stock) via `INSERT OR REPLACE`. Methods: `save()`, `findBySymbol()`, `deleteAllExcept(Set<String>)`. Streak deleted when signal stops firing.
-   **`SqliteNewlyAddedSymbolRepository`:** Queue table for symbols awaiting OHLCV backfill. `INSERT OR REPLACE` on add, `DELETE ... RETURNING` for atomic expired cleanup. Methods: `insert()`, `findOldest()`, `deleteAll()`, `deleteExpiredReturning()`.
-   **`SqliteApexPerformerRepository`:** Persists the apex performer set — stocks outperforming the top sector ETF (positive RS-vs-leader). Refreshed atomically (DELETE + batch INSERT in a transaction) by `SectorRelativeStrengthTracker.sendDailySectorRsSummary()` on its 16:00 / 21:00 CET cron. Read by `PullbackBuyTracker` to highlight buy alerts. Methods: `replaceAll(Set<String>)`, `findAll()`. Persisted (not in-memory) so the set survives the daily ~23:30 deploy window — required for Korean stock alerts that fire before the next refresh.
-   **`SqliteTreasuryIndicatorStateRepository`:** Per-series state for the FRED Treasury macro indicators tracked by `TreasuryTracker`. Stores one row per FRED series (`T10Y3M`, `T10Y2Y`, `DFII10`, `THREEFYTP10`) carrying the last classification band, raw value, observation date, and an `initialized` flag (matches the `momentum_roc_state` convention as a hatch for manually-seeded rows; production writes always set it to `true`). First-run alert suppression in `TreasuryTracker` keys off the absence of a row, not the flag. `INSERT OR REPLACE` on save. Methods: `save(TreasuryIndicatorState)`, `findBySeriesId(String)`. Survives bot restarts so we never silently miss a yield-curve inversion event mid-restart. Added in #516.
-   **`FeatureToggleService`:** Runtime feature flag management with JSON persistence. Loaded once via `@PostConstruct`, updated in-memory and on disk via `setToggle(FeatureToggle, boolean)`. Controlled at runtime via `/toggle` Telegram command. Creates the JSON file if missing; can enable toggles not yet present in the file (enum is the source of truth).
-   **`DatabaseDirectoryInitializer`:** `@PostConstruct` component that ensures the SQLite database parent directory exists at startup. Parses `spring.datasource.url` to extract the file path.
-   **Schema Management:** All DDL centralized in `src/main/resources/schema.sql` (13 tables). Auto-executed on startup via `spring.sql.init.mode=always`.

## Design Patterns

-   **Repository Pattern:** Interface-implementation separation. `PriceQuoteRepository`, `OhlcvRepository`, `MomentumRocRepository`. All implementations use Spring's `JdbcTemplate` (not raw JDBC). DataSource auto-configured via `spring.datasource.*` with HikariCP (pool size 1 for SQLite). Spring's `DataAccessException` propagates naturally (no manual `IllegalStateException` wrapping).
-   **Command Pattern**: Telegram command processing (`/add`, `/remove`, `/rsi`, `/show`, `/set`, `/data reset`, `/toggle`).
-   **Dependency Injection**: Constructor injection via `@RequiredArgsConstructor`. All major components injected.
-   **Profile Gating**: Default = production. `dev` = opt-in local profile.
-   **Scheduler Pattern**: `@Scheduled` with cron expressions and timezone support.
-   **Strategy Pattern**: Different `PriceEvaluator` implementations for different data sources.
-   **Facade Pattern**: `TelegramClient` simplifies Telegram Bot API interaction.
-   **Data Source Fallback**: `DailyPriceProvider` tries OHLCV first, falls back to Finnhub.
-   **Caching Decorator**: Repository-level cache implemented as a `@Primary` decorator over the SQLite implementation. Canonical example: `CachingOhlcvRepository` wraps `SqliteOhlcvRepository`. Consumers inject the interface (`OhlcvRepository`) and receive the decorator transparently — they don't know data came from cache vs. SQLite. The decorator owns invalidation: every write method (`saveAll`, `deleteBySymbol`) invalidates entries for the touched symbols. This pattern fits any read-heavy repository where the underlying data has clear write boundaries.
-   **Per-Reason TTL**: `IgnoreReason` enum carries `ttlSeconds` per value. `TargetPriceProvider.isSymbolIgnored()` uses `reason.getTtlSeconds()` instead of a global constant. Existing alerts use 12h, pullback uses 8h.
-   **Finnhub as Data Ingestion Layer**: `FinnhubPriceEvaluator` fetches live prices for ALL domestic tracked symbols (stocks + ETFs via `symbolRegistry.getAll()`, skipping international) and populates `LivePriceCache`. Loop 1 handles fetch/cache/persist/high-change-alerts for every symbol. Loop 2 evaluates target buy/sell prices from cache (no API calls). `YahooPriceEvaluator` does the same for international symbols. All downstream indicators/trackers read from `LivePriceCache` or SQLite — never make their own API calls.
-   **Phased Smoke Test**: `DevJobController.runAll()` executes 19 jobs in 4 dependency-ordered phases (seed → OHLCV fetch → parallel independents → VFI). Returns aggregate pass/fail with per-job results. `RootErrorHandler.runWithStatus()` provides boolean success/failure without propagating exceptions.
-   **Two-Pass OHLCV Fetch**: `OhlcvFetcher.fetchAndBackfillOhlcv()` runs domestic symbols first (TwelveData, 9s delay, retry + Telegram alert on failure), then international symbols (Yahoo Finance, 3s delay, log-only on failure). International detection via any-dot heuristic (`ticker.contains(".")`).
-   **Yahoo HTTP Transport**: `YahooFinanceClient.executeRequest()` uses `java.net.http.HttpClient` with `User-Agent: Mozilla/5.0`. The original `ProcessBuilder + curl` path was introduced under the assumption that Yahoo blocks Java HTTP clients via TLS fingerprinting — a hypothesis taken from the symptom that `RestTemplate` calls were rejected. ta4j evidence (#435) suggested this was actually a UA filter on `RestTemplate`'s default User-Agent, not TLS fingerprinting. The `HttpClient` path was added behind `FeatureToggle.YAHOO_HTTP_CLIENT` for production verification, ran successfully for ~2 weeks with no SSL/TLS failures, and the legacy curl path was removed in #457. Per-request timeout 15s, connect timeout 10s; on non-2xx the exception message includes status code + headers + body; on I/O failure it includes the exception class simple name (e.g. `ConnectException`) for transport-level failure visibility without DEBUG logging.
-   **Secret Redaction in Logs (Defense-in-Depth)**: Two-layer approach following the #464 incident. Layer 1: don't put secret-bearing data in egress paths with broad blast radius — `RootErrorHandler` no longer routes exception details to Telegram (#469). Layer 2: a global `SecretRedactingTurboFilter` registered in `logback-spring.xml` drops any log event whose format / params / throwable cause chain matches a known secret shape (#470). Fail-closed by design: better to lose a diagnostic line than leak a credential. Pattern is `static final List<Pattern>` so additional secret shapes (e.g., API keys for new providers) can be added with one line. Pattern source: trufflehog's industry-standard regex.
-   **Configurable Lookback + Min-Data-Points Pair**: Convention for any service that consumes `DailyPriceProvider.findDailyClosingPrices(symbol, days)`. The `days` parameter is interpreted by the repo as a *calendar* window (`WHERE date >= date('now', '-N days')`), not a row count. Callers must oversize the calendar window so the query returns at least their required row count year-round (~5/7 trading-day yield, minus market holidays). Each caller pairs `LOOKBACK_CALENDAR_DAYS` (or analogous constant) with a separate `MIN_DATA_POINTS` (or `MIN_HISTORY_SIZE`, `period+1`, etc.) used as the actual statistical threshold. EmaService/VfiService/RelativeStrengthService follow this; `TailRiskService` was the deviant until #472 introduced `TailRiskWindow(lookbackCalendarDays, minDataPoints)` to couple the two values into one type. New callers should follow the same pattern; conflating fetch-window with min-row-count produces unsatisfiable validation.
-   **Bias-Corrected Sample Estimators for Statistical Moments**: When computing statistical moments (kurtosis, skewness, and any higher-order moment added later), use bias-corrected sample estimators — specifically Fisher-Pearson G1 (skewness) and G2 (kurtosis) — delegated to `org.apache.commons.statistics.descriptive`. These are the industry standard in quantitative finance: Pandas, NumPy (with `bias=False`), Excel `KURT`/`SKEW`, R `e1071::kurtosis(type=2)`, and the Basel-FRTB framework all default to them. The biased method-of-moments alternatives (`m₄/m₂²`, `m₃/m₂^(3/2)`) systematically under-read at small n (~8% downward bias on kurtosis at n=25, ~15% on skewness magnitude) — the wrong direction of error for risk metrics, since under-reading hides tail risk. Established in #433 after `TailRiskService` originally shipped with the biased formulas. New indicators that compute higher-order moments should default to Commons Statistics' factory API (`Kurtosis.of(double[]).getAsDouble()` etc.) rather than rolling their own formulas. Preserve `Double.NaN` safety guards (size checks + zero-variance) at the call site — Commons Statistics returns `NaN` for those edge cases, and downstream level classifiers (`fromExcessKurtosis(NaN)` etc.) misclassify NaN as EXTREME because `NaN < 1.0` falls through all the comparison branches.
-   **External Macro Data Provider (free, low-volume)**: Pattern for consuming a free external macro data source that doesn't need metering. `EnricoClient` (Kayaposoft public-holidays, no auth) and `FredClient` (FRED, single API-key query param) both follow this shape: a single Spring `@Component`, a single read method returning `List<DomainRecord>` (or `Map`), wire DTOs as records with `@JsonIgnoreProperties(ignoreUnknown = true)`, empty-collection-plus-WARN on every failure mode, no `ApiRequestMeteringService` integration. The justification: 4 requests/day to FRED + ~5 startup calls to Enrico are below the noise floor of the metering reports. New macro/economic-data providers should mirror this. Counter-example: `TwelveDataClient` and `YahooFinanceClient` are metered because they fetch per-symbol data with quota constraints — that's a different pattern.
-   **Symbol Lifecycle Listener**: Components owning ticker-keyed SQLite state implement `SymbolLifecycleListener.onSymbolRemoved(String ticker)`. `SymbolManagementService.removeSymbol` injects `List<SymbolLifecycleListener>` and fans out after registry + target-price cleanup. Fan-out is unordered and fail-isolated (per-listener try/catch → `log.warn`). Adopted in #523 to fix orphaned per-symbol rows after a remove/re-add cycle (RSI/RS/ROC state, OHLCV cache, accumulation streaks, ignored-symbols flags, etc.). When adding a new ticker-keyed table, make its owning component implement the interface — no further wiring required. **Documented inconsistency**: `target_prices` and `tracked_symbols` cleanup runs via the legacy direct-call path in `SymbolManagementService` (those calls predate the listener pattern and have non-void return contracts); every other ticker-keyed table goes through the listener. **OHLCV special case**: `CachingOhlcvRepository` (not the underlying `SqliteOhlcvRepository`) is the listener so cache invalidation rides along — making both classes listeners would fire the DELETE twice. End-user feedback is intentionally unchanged: cleanup is invisible from Telegram and the dashboard; failures are operator-visible via logs only. Idempotence comes from SQL DELETE atomicity + `SymbolRegistry.removeSymbol` returning `true` exactly once across concurrent callers — no Java-level synchronization needed.

## Delete-Before-Send Pattern (Telegram)

For recurring hourly reports (BB, RSI), the bot deletes the previous message before sending the new one:
- In-memory `lastTelegramReportMessageId` tracking
- `sendMessageAndReturnId()` returns `OptionalLong`
- Delete failures logged but don't block new message delivery

Daily reports (VFI, EMA, Tail Risk, BB daily) use regular `sendMessage()` — each day's report stays in chat.

## Testing Patterns

- **Mock-based testing**: External dependencies mocked with Mockito
- **Argument Captors**: For verifying complex method arguments
- **Temp Files**: `@TempDir` for file persistence tests
- **Repository Slice Tests**: `@JdbcTest` + `@AutoConfigureTestDatabase(replace = NONE)` + `@Sql("classpath:schema.sql")`. Spring Boot 4 package: `org.springframework.boot.jdbc.test.autoconfigure`. Uses Spring-managed in-memory SQLite with transactional rollback per test.
- **Plain JUnit + JdbcTemplate**: Tests constructing repositories directly use file-based temp SQLite DBs with manual schema init via `JdbcTemplate.execute()`.
- **Configurable File Paths**: Constructor injection for file paths in tests
- **Lenient stubs**: Used when SymbolRegistry methods return real ETF data via static constants but only specific symbols are relevant to the test

## Dashboard Frontend Patterns

### SSEProvider (Single EventSource)
- One `EventSource` per app lifetime, owned by `SSEProvider`
- Components subscribe via `useSSE().subscribe(type, handler)` — never open their own `EventSource`
- Auth cookie/token attaches here when auth lands
- `SSEContext` exported for test injection via `<SSEContext.Provider value={mock}>`

### State Ownership Rule
- Per-browser preferences (theme, layout) → `localStorage`
- Per-user state (watchlist, selectedSymbols) → server-side once auth lands; never `localStorage`
